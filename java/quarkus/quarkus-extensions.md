## Summary

Everything in Quarkus is an extension — from the REST layer to database access. Extensions participate in Quarkus's build-time processing pipeline, providing automatic native-image support and dev-mode integration. This doc covers the most important extensions: [Hibernate ORM with Panache](https://quarkus.io/guides/hibernate-orm-panache) (simplified JPA), [MicroProfile REST Client](https://quarkus.io/guides/rest-client) (type-safe HTTP), [Qute](https://quarkus.io/guides/qute) (templating), [SmallRye](https://smallrye.io/) implementations of MicroProfile specs, the Spring compatibility layer, and the [Quarkiverse](https://github.com/quarkiverse) community ecosystem.

---

## How Extensions Work

A Quarkus extension has two parts:

1. **Build-time module** (`deployment`) — runs during `mvn package`; scans classes, generates bytecode, registers native-image config
2. **Runtime module** (`runtime`) — the code that ships in the final application

This split means extensions can perform heavy work at build time (reflection registration, proxy generation, config validation) so the runtime is lean. When you add `quarkus-hibernate-orm-panache`, its deployment module:

- Scans for `@Entity` classes via Jandex
- Generates accessor methods for public fields
- Registers all entities for reflection (native-image)
- Validates persistence configuration

At runtime, only the pre-generated bytecode runs — no startup scanning.

```bash
# Add extensions
quarkus extension add hibernate-orm-panache jdbc-postgresql rest-jackson

# List installed extensions
quarkus extension list --installed
```

## Panache — Simplified Hibernate ORM

[Hibernate ORM with Panache](https://quarkus.io/guides/hibernate-orm-panache) is Quarkus's opinionated layer on top of JPA/Hibernate, inspired by Ruby on Rails's Active Record and Spring Data's repository pattern. It eliminates boilerplate while keeping full JPA access when needed.

### Active Record Pattern

Entities extend `PanacheEntity` and get CRUD methods as static methods:

```java
import io.quarkus.hibernate.orm.panache.PanacheEntity;
import jakarta.persistence.Entity;

@Entity
public class Fruit extends PanacheEntity {
    // No @Id needed — PanacheEntity provides auto-generated Long id
    // Use public fields — Panache generates getters/setters at build time
    public String name;
    public String color;
    public int calories;
}
```

Usage — no repository class needed:

```java
// Find all
List<Fruit> fruits = Fruit.listAll();
List<Fruit> fruits = Fruit.listAll(Sort.by("name"));

// Find by id
Fruit apple = Fruit.findById(1L);

// Find by query
List<Fruit> reds = Fruit.list("color", "red");
Fruit first = Fruit.find("name", "Apple").firstResult();

// Persist
Fruit mango = new Fruit();
mango.name = "Mango";
mango.color = "yellow";
mango.persist();

// Delete
Fruit.deleteById(1L);
long deleted = Fruit.delete("color", "green");

// Count
long count = Fruit.count();
long redCount = Fruit.count("color", "red");

// Pagination
PanacheQuery<Fruit> query = Fruit.findAll();
query.page(Page.of(0, 25));
List<Fruit> firstPage = query.list();
```

### Repository Pattern

If you prefer separating persistence from the entity:

```java
import io.quarkus.hibernate.orm.panache.PanacheRepository;
import jakarta.enterprise.context.ApplicationScoped;

@ApplicationScoped
public class FruitRepository implements PanacheRepository<Fruit> {
    // All PanacheEntity static methods are now instance methods
    // findById, listAll, persist, delete, count, etc.

    public List<Fruit> findByColor(String color) {
        return list("color", color);
    }

    public Fruit findByName(String name) {
        return find("name", name).firstResult();
    }
}
```

Usage:

```java
@Inject
FruitRepository fruitRepo;

List<Fruit> reds = fruitRepo.findByColor("red");
fruitRepo.persist(newFruit);
```

### Active Record vs Repository

| Aspect | Active Record (`extends PanacheEntity`) | Repository (`implements PanacheRepository`) |
|--------|---------------------------------------|-------------------------------------------|
| Entity coupling | Entity has persistence methods | Entity is a plain POJO |
| Testing | Mock static methods (harder) | Inject and mock repository (easier) |
| Separation of concerns | Entity = data + persistence | Entity = data, repository = persistence |
| Boilerplate | Less (no separate class) | More (separate repo class) |
| Best for | Simple CRUD, rapid prototyping | Complex business logic, clean architecture |

### Custom Queries

```java
// HQL/JPQL
List<Fruit> results = Fruit.find(
    "SELECT f FROM Fruit f WHERE f.calories < ?1 AND f.color = ?2",
    200, "red"
).list();

// Named parameters
List<Fruit> results = Fruit.find(
    "calories < :maxCal AND color = :color",
    Parameters.with("maxCal", 200).and("color", "red")
).list();

// Native SQL
List<Fruit> results = Fruit.find(
    "#Fruit.findLowCal",  // references @NamedQuery
    Parameters.with("maxCal", 200)
).list();
```

### Spring Data JPA Comparison

| Spring Data JPA | Panache Active Record | Panache Repository |
|----------------|----------------------|-------------------|
| `findAll()` | `Fruit.listAll()` | `repo.listAll()` |
| `findById(id)` | `Fruit.findById(id)` | `repo.findById(id)` |
| `save(entity)` | `entity.persist()` | `repo.persist(entity)` |
| `deleteById(id)` | `Fruit.deleteById(id)` | `repo.deleteById(id)` |
| Derived queries (`findByColor`) | Not supported — write HQL | Custom method with `list()` |
| `@Query("...")` | `Fruit.find("...")` | `repo.find("...")` |
| Specifications | Not built-in | Not built-in |

Panache is simpler but less feature-rich than Spring Data JPA. No derived query methods, no Specifications, no `@Query` annotation on repository methods. For complex queries, you write HQL/JPQL strings.

## REST Client — Type-Safe HTTP

[MicroProfile REST Client](https://quarkus.io/guides/rest-client) generates HTTP clients from annotated interfaces — similar to Spring Cloud OpenFeign:

```java
import org.eclipse.microprofile.rest.client.inject.RegisterRestClient;
import jakarta.ws.rs.GET;
import jakarta.ws.rs.Path;
import jakarta.ws.rs.PathParam;

@RegisterRestClient(configKey = "user-api")
@Path("/users")
public interface UserApiClient {

    @GET
    @Path("/{id}")
    User getUser(@PathParam("id") long id);

    @GET
    List<User> listUsers();
}
```

Configuration:

```properties
quarkus.rest-client.user-api.url=http://user-service:8080
quarkus.rest-client.user-api.scope=jakarta.inject.Singleton
```

Injection:

```java
import org.eclipse.microprofile.rest.client.inject.RestClient;

@Inject
@RestClient
UserApiClient userApi;
```

Reactive variant — return `Uni<T>`:

```java
@RegisterRestClient(configKey = "user-api")
@Path("/users")
public interface UserApiClientReactive {

    @GET
    @Path("/{id}")
    Uni<User> getUser(@PathParam("id") long id);
}
```

## Qute — Templating Engine

[Qute](https://quarkus.io/guides/qute) is Quarkus's templating engine for server-rendered HTML, emails, and text output. It validates templates at build time.

```html
<!-- src/main/resources/templates/fruits.html -->
<!DOCTYPE html>
<html>
<body>
    <h1>Fruits ({fruits.size})</h1>
    <ul>
        {#for fruit in fruits}
        <li>{fruit.name} — {fruit.calories} cal</li>
        {/for}
    </ul>
    {#if fruits.isEmpty}
    <p>No fruits found.</p>
    {/if}
</body>
</html>
```

```java
import io.quarkus.qute.Template;
import io.quarkus.qute.TemplateInstance;
import jakarta.inject.Inject;

@Path("/fruits")
public class FruitResource {

    @Inject
    Template fruits;  // matches templates/fruits.html by field name

    @GET
    @Produces(MediaType.TEXT_HTML)
    public TemplateInstance list() {
        return fruits.data("fruits", Fruit.listAll());
    }
}
```

Build-time validation catches typos in template expressions — `{fruit.naem}` fails the build if `Fruit` has no `naem` field.

## SmallRye Implementations

Quarkus uses [SmallRye](https://smallrye.io/) for MicroProfile spec implementations:

| MicroProfile Spec | SmallRye Implementation | Quarkus Extension |
|-------------------|----------------------|-------------------|
| Config | SmallRye Config | `quarkus-arc` (built-in) |
| Health | SmallRye Health | `quarkus-smallrye-health` |
| Metrics | Micrometer (replaces SmallRye Metrics) | `quarkus-micrometer` |
| Fault Tolerance | SmallRye Fault Tolerance | `quarkus-smallrye-fault-tolerance` |
| OpenAPI | SmallRye OpenAPI | `quarkus-smallrye-openapi` |
| Reactive Messaging | SmallRye Reactive Messaging | `quarkus-smallrye-reactive-messaging-kafka` |
| JWT Auth | SmallRye JWT | `quarkus-smallrye-jwt` |
| GraphQL | SmallRye GraphQL | `quarkus-smallrye-graphql` |

These work identically to the MicroProfile APIs covered in [Helidon MP](../helidon/helidon-mp.md) — `@Retry`, `@CircuitBreaker`, `@Liveness`, `@ConfigProperty`, etc. The annotations are the same; the runtime is different.

## Spring Compatibility Layer

Quarkus provides `quarkus-spring-*` extensions that implement Spring API subsets using Quarkus's build-time engine:

```bash
quarkus extension add spring-di spring-web spring-data-jpa
```

| Extension | What It Supports |
|-----------|-----------------|
| `quarkus-spring-di` | `@Autowired`, `@Component`, `@Service`, `@Repository`, `@Configuration`, `@Bean` |
| `quarkus-spring-web` | `@RestController`, `@GetMapping`, `@PostMapping`, `@RequestBody`, `@PathVariable` |
| `quarkus-spring-data-jpa` | `CrudRepository`, `PagingAndSortingRepository`, derived query methods |
| `quarkus-spring-security` | `@Secured`, `@RolesAllowed` |
| `quarkus-spring-boot-properties` | `@ConfigurationProperties` |
| `quarkus-spring-cache` | `@Cacheable`, `@CacheEvict` |
| `quarkus-spring-scheduled` | `@Scheduled` |

**Important caveats:**

- These extensions compile Spring annotations into Quarkus-native code at build time — they do not include the Spring Framework runtime
- Only a **subset** of each Spring API is supported — advanced features (SpEL in `@Value`, `@Conditional*`, `BeanPostProcessor`) are not available
- The goal is **migration assistance**, not full Spring compatibility. New Quarkus projects should use native Quarkus/Jakarta APIs.

## Quarkiverse — Community Extensions

[Quarkiverse](https://github.com/quarkiverse) is the community extension ecosystem, hosted under a dedicated GitHub org with standardized CI, release process, and quality gates. Extensions include:

- **quarkus-langchain4j** — LLM integration (OpenAI, Ollama, Anthropic)
- **quarkus-amazon-services** — AWS SDK (S3, SQS, DynamoDB, etc.)
- **quarkus-google-cloud-services** — GCP SDK
- **quarkus-minio** — MinIO object storage
- **quarkus-quinoa** — frontend integration (React, Angular, Vue dev servers)
- **quarkus-bucket4j** — rate limiting
- **quarkus-jooq** — jOOQ integration
- **quarkus-rabbitmq-client** — RabbitMQ client

Browse all extensions at [quarkus.io/extensions](https://quarkus.io/extensions/) or via `quarkus extension list --search <keyword>`.
