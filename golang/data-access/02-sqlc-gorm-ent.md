## Summary

[`database/sql`](01-database-sql.md) gives you a pool, a cursor, and a transaction API. Anything beyond positional `Scan` calls is third-party. Three libraries dominate the next layer in the Go data-access stack, each occupying a distinct point on the abstraction spectrum:

- **sqlc** â€” write SQL, generate strongly-typed Go functions. No runtime reflection. No query builder. SQL is the source of truth.
- **GORM** â€” full Active-Record-flavoured ORM. Auto-migrations, hooks, associations, soft deletes. Heavy reflection at runtime.
- **Ent** â€” schema-as-Go-code, generates a fluent typed query builder. Originated at Facebook/Meta, now maintained by the Atlas team. Strong on graph traversals.

There is no "winner". The choice is about how much SQL your team wants to write, how much magic you can tolerate, and where you want surprises to land â€” at compile time (sqlc, Ent) or at runtime (GORM). This doc walks through each, then compares them with Prisma (TypeScript) and JPA / Hibernate (Java).

## 1. Three philosophies in one ecosystem

| | sqlc | GORM | Ent |
|---|---|---|---|
| Source of truth | `.sql` query files | Go structs (tags) | Go schema files |
| Generates code | Yes | No (runtime reflection) | Yes |
| Query API style | Generated functions | Method chaining | Fluent typed builder |
| Relations | Manual JOINs | Associations + Preload | Edges + traversal |
| Migrations | External tool | `AutoMigrate()` or external | Built-in versioned (Atlas) |
| Active Record-ish | No | Yes | Partially (entity types) |
| Reflection at runtime | None | Heavy | None |
| TypeScript analogue | (closest) Drizzle / kysely-codegen | Prisma's runtime feel | Prisma's schema-first feel |
| Java analogue | jOOQ / MyBatis | (none â€” JPA is closer to Ent) | JPA / Hibernate |

If you grew up on Prisma, **sqlc** is the most surprising â€” there is no schema file that becomes Go types. You hand-write SQL, and a CLI tool generates `db.GetUser(ctx, id) (User, error)` functions matching each query's columns and parameters. **GORM** is the most familiar â€” it looks like Sequelize or TypeORM. **Ent** is the most Prisma-shaped â€” declarative schema, generated client, typed queries â€” but with Go's compile-time guarantees.

## 2. sqlc â€” write SQL, generate Go

The pitch is one line: SQL is the API. You write `.sql` files with annotated comments naming each query and saying what shape it returns. `sqlc` parses them, type-checks against your schema, and generates Go functions.

A query file:

```sql
-- queries/users.sql

-- name: GetUser :one
SELECT id, email, created_at FROM users WHERE id = $1;

-- name: ListActiveUsers :many
SELECT id, email, created_at
FROM users
WHERE last_login_at > $1
ORDER BY created_at DESC
LIMIT $2;

-- name: CreateUser :one
INSERT INTO users (email, password_hash)
VALUES ($1, $2)
RETURNING id, email, created_at;

-- name: DeactivateInactiveUsers :exec
UPDATE users SET active = false
WHERE last_login_at < $1;
```

The annotations:
- `:one` â€” returns a single row (or `sql.ErrNoRows`).
- `:many` â€” returns a slice.
- `:exec` â€” returns `sql.Result` (rows affected, etc.).
- `:execrows` â€” returns affected row count directly.
- `:batchone` / `:batchmany` / `:batchexec` â€” pgx batched variants.

You define the schema in a separate file (or point sqlc at your migrations):

```sql
-- schema.sql
CREATE TABLE users (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    email TEXT NOT NULL UNIQUE,
    password_hash TEXT NOT NULL,
    active BOOLEAN NOT NULL DEFAULT true,
    last_login_at TIMESTAMPTZ,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

A `sqlc.yaml` ties them together:

```yaml
version: "2"
sql:
  - engine: "postgresql"
    queries: "queries/"
    schema: "schema.sql"
    gen:
      go:
        package: "db"
        out: "internal/db"
        sql_package: "pgx/v5"
```

Run `sqlc generate`. You get:

```go
// generated by sqlc
package db

type User struct {
    ID           pgtype.UUID
    Email        string
    PasswordHash string
    Active       bool
    LastLoginAt  pgtype.Timestamptz
    CreatedAt    pgtype.Timestamptz
}

func (q *Queries) GetUser(ctx context.Context, id pgtype.UUID) (User, error) {
    row := q.db.QueryRow(ctx, getUser, id)
    var u User
    err := row.Scan(&u.ID, &u.Email, &u.CreatedAt)
    return u, err
}

func (q *Queries) ListActiveUsers(ctx context.Context, lastLoginAt pgtype.Timestamptz, limit int32) ([]User, error) {
    // ...
}
```

Call site:

```go
queries := db.New(pool)  // pool is *pgxpool.Pool
user, err := queries.GetUser(ctx, userID)
if err != nil {
    if errors.Is(err, pgx.ErrNoRows) {
        return nil, ErrUserNotFound
    }
    return nil, err
}
```

**What you get:**
- Compile-time errors for any drift between SQL and Go (column added, type changed, query renamed).
- No runtime reflection. The generated code is what you would write by hand.
- Real Postgres types via `pgtype` (when targeting `pgx/v5`).
- IDE jump-to-definition lands on the generated `.go` file, which contains the exact SQL string. No mystery as to what runs.

**What you don't get:**
- No migrations. Bring `golang-migrate`, `goose`, or `Atlas`.
- No relations. If you need to join users to their posts, you write the JOIN in SQL.
- No runtime query building. If you need dynamic `WHERE` clauses, you either write all the variants or drop to raw SQL.
- No schema introspection beyond what's in `schema.sql`.

The "no relations" point is fundamental. Coming from Prisma's `include: { posts: true }`, sqlc feels minimalistic. The trade is: you always know the SQL plan, you never accidentally fetch a 10-MB relation, you never get N+1 from lazy loading, but you write more SQL.

**Database support:** Postgres (best supported), MySQL, SQLite. Code generation for Go, plus Kotlin, Python, and TypeScript via plugins.

## 3. GORM â€” full ORM with reflection

GORM is the most popular full-featured ORM in the Go ecosystem. It does what you'd expect a TypeScript or Java developer to want:

```go
type User struct {
    ID        uint      `gorm:"primaryKey"`
    Email     string    `gorm:"uniqueIndex;not null"`
    Posts     []Post    // has-many
    CreatedAt time.Time
    UpdatedAt time.Time
    DeletedAt gorm.DeletedAt `gorm:"index"`  // soft delete
}

type Post struct {
    ID        uint
    Title     string
    Content   string
    UserID    uint
    User      User      // belongs-to
    CreatedAt time.Time
}

// Auto-create tables from struct definitions
db.AutoMigrate(&User{}, &Post{})

// Insert
user := User{Email: "alice@example.com"}
db.Create(&user)  // populates user.ID

// Query with eager loading
var users []User
db.Preload("Posts").Where("created_at > ?", lastWeek).Find(&users)

// Update
db.Model(&user).Update("Email", "alice@new.com")

// Soft delete (sets DeletedAt instead of DELETE)
db.Delete(&user)
```

**Strengths:**
- Familiar. If you've used Sequelize, TypeORM, or Active Record, you're productive in an hour.
- Auto-migrations from struct definitions get you started fast.
- Hooks (`BeforeCreate`, `AfterUpdate`, etc.), associations (has-many, many-to-many, polymorphic), soft deletes, batch operations, transactions, and a query builder all work out of the box.
- Plugin API (sharding, encryption, audit logs).

**Criticisms (consistent across the community):**
- **Reflection cost.** Every query reflects over the struct to build the SQL and scan the result. For services where the database is the bottleneck, this is fine. For services where Go-side overhead matters, sqlc and Ent measurably outperform.
- **String-based field references.** `db.Where("created_at > ?", lastWeek)` is a string. A typo in the column name fails at runtime. `db.Preload("Posts")` is also a string. Refactoring is fragile.
- **Footguns.** The chainable API can silently do the wrong thing. `db.First(&user, id)` errors with `ErrRecordNotFound`; `db.Find(&user, id)` does not (zero rows is success). `Preload` with a malformed condition silently fetches nothing. The error model has improved over versions but still surprises new users.
- **AutoMigrate is not a real migration tool.** It can add columns and indexes; it will not drop columns, change types safely, or produce a migration history. Production teams disable it and use `golang-migrate` / `goose` / `Atlas` instead.

**When GORM is the right call:**
- Fast prototyping where developer ergonomics outweigh runtime cost.
- Teams from a Rails / Django / Laravel background where Active Record patterns are deeply familiar.
- CRUD-heavy services where the runtime overhead is a small fraction of total request time.

## 4. Ent â€” schema-as-code with a fluent typed builder

Ent originated at Facebook/Meta as an internal entity framework, was open-sourced as `entgo.io`, and is now developed by the Atlas team (the same group behind the Atlas migration tool). It is closest in spirit to Prisma: a declarative schema becomes a typed client.

The schema is **Go code**, not a DSL:

```go
// ent/schema/user.go
package schema

import (
    "entgo.io/ent"
    "entgo.io/ent/schema/edge"
    "entgo.io/ent/schema/field"
)

type User struct {
    ent.Schema
}

func (User) Fields() []ent.Field {
    return []ent.Field{
        field.UUID("id", uuid.UUID{}).Default(uuid.New),
        field.String("email").Unique().NotEmpty(),
        field.Bool("active").Default(true),
        field.Time("created_at").Default(time.Now).Immutable(),
    }
}

func (User) Edges() []ent.Edge {
    return []ent.Edge{
        edge.To("posts", Post.Type),  // has-many
    }
}

func (User) Indexes() []ent.Index {
    return []ent.Index{
        index.Fields("active", "created_at"),
    }
}
```

Run `go generate ./ent`. Ent produces a typed client:

```go
client, err := ent.Open("postgres", dsn)

// Create
user, err := client.User.Create().
    SetEmail("alice@example.com").
    Save(ctx)

// Query
users, err := client.User.Query().
    Where(user.ActiveEQ(true), user.CreatedAtGT(lastWeek)).
    Order(ent.Desc(user.FieldCreatedAt)).
    Limit(20).
    All(ctx)

// Edge traversal â€” fully typed
posts, err := client.User.Query().
    Where(user.ID(userID)).
    QueryPosts().
    Where(post.PublishedEQ(true)).
    All(ctx)
```

Every field name, every operator, every edge is a generated typed function. `user.ActiveEQ`, `user.FieldCreatedAt`, `QueryPosts` â€” all generated. A typo doesn't compile.

**Strengths:**
- **Compile-time-checked queries** without writing SQL. Best of both worlds versus sqlc and GORM.
- **Graph traversal is first-class.** `client.User.Query().QueryPosts().QueryComments().QueryAuthor()` chains across edges with full typing. This is what Facebook used internally and what gives Ent its identity.
- **Built-in versioned migrations** via Atlas integration. `atlas migrate diff` reads the Ent schema and produces migration SQL.
- **Schema-time hooks, mixins, policies** for cross-cutting concerns (audit logging, soft delete, multi-tenancy).
- **No runtime reflection on hot paths.** The generated code uses concrete types end-to-end.

**Trade-offs:**
- The schema language is Go code with a lot of boilerplate methods. More verbose than Prisma's `.prisma` DSL.
- Smaller community than GORM. Fewer Stack Overflow answers, fewer plugins.
- Code generation step is mandatory in the build. Schema changes require `go generate` before you can call new methods.
- Heavy generated output (many files per entity). Initial codebase size is larger than sqlc.

**When Ent is the right call:**
- Domain models with rich relationships and graph queries (social networks, knowledge graphs, anything where traversal matters).
- Teams that want Prisma-like ergonomics with Go's compile-time guarantees.
- Projects that want migrations integrated with the schema (Atlas + Ent is the closest Go gets to Prisma's `migrate dev` story).

## 5. Decision matrix

A practical chooser:

| If you wantâ€¦ | Use |
|---|---|
| Maximum control over SQL, no surprises, every query visible | **sqlc** |
| Fast prototyping, Active Record ergonomics, CRUD-heavy app | **GORM** |
| Schema-first, typed query builder, rich relationship traversal | **Ent** |
| To migrate from Prisma to Go and keep similar ergonomics | **Ent** |
| To migrate from Hibernate to Go and keep `EntityManager`-ish patterns | **GORM** (closest mental model) or **Ent** (better long-term fit) |
| To avoid reflection and binary bloat, ship in serverless / edge | **sqlc** |
| Small team, mostly raw SQL already, just want type-safe scans | **sqlc** with a thin `pgx` repository layer |
| Need dynamic queries (search filters, configurable reports) | **Ent** or **GORM**; sqlc is awkward here |

A few common patterns I've seen in production Go services in 2026:

- **`pgx` (native) + `sqlc` + Atlas.** Common new-build pick. All SQL visible, full Postgres types, no reflection. Atlas handles migrations.
- **`pgx` + GORM, AutoMigrate disabled, `golang-migrate` for schema.** Teams that liked GORM's API but learned not to trust AutoMigrate.
- **`pgx` + Ent + Atlas.** Teams with Prisma / Hibernate backgrounds and complex relational models.
- **`database/sql` + `sqlx` + `golang-migrate`.** The minimal stack. No code generation, modest reflection from sqlx, simple migrations.

## 6. Migrations per tool

| Tool | Migration story | Versioned files | Schema source of truth |
|---|---|---|---|
| sqlc | External (`golang-migrate` / `goose` / Atlas) | Yes (whatever you choose) | `schema.sql` consumed by sqlc |
| GORM | `AutoMigrate(&User{})` for prototyping; external for production | Optional (external tool) | Go struct tags |
| Ent | Built-in via Atlas: `atlas migrate diff` reads Ent schema | Yes | Ent Go schema |

**sqlc** is migration-agnostic. It needs to know your schema (to type-check queries) but does not generate migrations. Pair it with whatever versioned migration tool you like â€” the recommended combo is `golang-migrate` for simple cases or `Atlas` for declarative diffing.

**GORM**'s `AutoMigrate` is "DDL synthesis" â€” it inspects struct tags and applies `CREATE TABLE`, `ALTER TABLE` to add missing columns/indexes. It will not drop columns or rename them, will not write migration history files, and gives you no rollback path. For production, disable it and use a real migration tool.

```go
// fine for tests / dev prototype
db.AutoMigrate(&User{}, &Post{})

// production: don't do this; run golang-migrate or Atlas before app boot
```

**Ent** integrates directly with Atlas. Atlas reads the Ent schema (Go code) and generates migration SQL files based on the diff against the live database. This gives you Prisma-`migrate dev`-style ergonomics with versioned files and a full SQL audit trail.

```bash
atlas migrate diff add_user_avatar \
    --to "ent://ent/schema" \
    --dev-url "docker://postgres/15/dev" \
    --dir "file://migrations"
```

## 7. Comparison: Prisma and JPA / Hibernate

A lateral view across ecosystems:

| Concern | Prisma (TS) | JPA / Hibernate (Java) | sqlc | GORM | Ent |
|---|---|---|---|---|---|
| Schema source | `.prisma` DSL | `@Entity` classes | `schema.sql` | Go struct tags | Go schema files |
| Generated client | Yes (Rust engine + JS) | No (runtime proxies) | Yes (pure Go) | No (reflection) | Yes (pure Go) |
| Type-safe queries | Yes | JPQL is string-based | Yes | Strings (runtime) | Yes (compile-time) |
| Relation loading | `include` / `select` | `FetchType.LAZY/EAGER` | Manual JOIN | `Preload` | Edge traversal |
| Migrations | `prisma migrate` | Flyway / Liquibase | External tool | `AutoMigrate` (don't) or external | Atlas (built-in) |
| Soft delete | extension | `@SQLDelete` / `@Where` | Manual | Built-in (`gorm.DeletedAt`) | Mixin |
| Optimistic locking | Manual `version` field | `@Version` | Manual | `optimisticlock` plugin | Manual |
| Identity map / dirty tracking | No | Yes (1st-level cache) | No | Partial (sessions) | No |
| Runtime reflection | Engine-side | Heavy | None | Heavy | None |

**Key mental shifts coming from Prisma:**

- The Go ecosystem accepts manual SQL more readily. sqlc is a popular pick precisely because it embraces it. Prisma users sometimes find this jarring; it is the norm here.
- There is no Prisma-style `prisma generate` triggered by editing a `.prisma` file. With Ent, you edit Go schema files and run `go generate`; with sqlc, you edit SQL and run `sqlc generate`. The trigger is explicit.
- "Connection pooler in front of the database" (PgBouncer) is more visible in Go discussions because the standard library exposes pool tuning directly. Prisma hides this until you hit it.

**Key mental shifts coming from JPA:**

- No `EntityManager`. No automatic dirty checking. Every UPDATE is a query you wrote (sqlc) or a method call that produces a query you can predict (Ent, GORM). Lost magic, gained predictability.
- No L1 cache. If you fetch the same row twice in a transaction, you get two queries. The JPA optimisation of "just call `find` again, the EntityManager has it" does not apply.
- No JPQL. The Go equivalents either accept SQL strings (GORM, sqlc) or build typed AST programmatically (Ent).
- `@Transactional` translates roughly to passing a `*sql.Tx` (or driver-specific equivalent) through the call chain. There is no aspect-oriented transaction propagation; the boundary is explicit. See [database/sql Â§7](01-database-sql.md#7-transactions-and-the-deferred-rollback-idiom).

## 8. Honourable mentions

A few other libraries that come up:

**[sqlboiler](https://github.com/volatiletech/sqlboiler).** Database-first code generator: introspects an existing schema and generates Go models. Conceptually similar to sqlc but inverted â€” sqlc parses your queries, sqlboiler generates a query builder for whatever the schema offers. **Now in maintenance mode**; the maintainers recommend [Bob](https://github.com/stephenafamo/bob) or sqlc for new projects.

**[bun](https://github.com/uptrace/bun).** "SQL-first" ORM / query builder by Uptrace (the APM vendor). Lightweight, supports Postgres, MySQL, SQLite, MSSQL, Oracle. Sits between sqlc (no runtime API at all) and GORM (full ORM): you get a typed query builder without code generation, with reflection used only for struct mapping. A good pick if you want a builder API but find Ent too heavy.

**[gorp](https://github.com/go-gorp/gorp).** Minimal struct-to-table mapper. Older project; useful when you want struct scanning slightly above `database/sql` without the weight of GORM. Less actively developed than `sqlx`; most teams pick `sqlx` instead.

**[Bob](https://github.com/stephenafamo/bob).** Newer query builder + ORM hybrid. Recommended successor to sqlboiler. Notable but smaller community than the three main options above.

For new Go services as of 2026, **sqlc**, **Ent**, and **GORM** are the three you weigh against each other. Bun is a credible fourth if you want a builder API specifically. Everything else is either niche or fading.
